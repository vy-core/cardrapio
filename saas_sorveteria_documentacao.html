Seção 1
Minimundo
Descrição do domínio do problema e contexto do negócio

Uma sorveteria/pizzaria de bairro deseja digitalizar seu processo de pedidos, que hoje acontece por telefone. O estabelecimento possui um cardápio de produtos (sorvetes, pizzas e complementos) que precisam ser exibidos online para que clientes possam fazer pedidos sem sair de casa.

O cliente acessa o sistema pelo navegador, navega pelo cardápio, monta seu pedido, informa seus dados de entrega e finaliza. Nenhum cadastro ou login é necessário — a experiência deve ser simples e rápida. O acompanhamento do status é feito no próprio navegador via localStorage.

Do lado do estabelecimento, um ou mais funcionários acessam uma área administrativa protegida por senha onde visualizam os pedidos em tempo real, atualizam seus status e obtêm as informações necessárias para preparo e entrega. Não há integração com meios de pagamento no MVP — todos os pagamentos ocorrem na entrega.

Atores do Sistema

Cliente — acesso público, sem autenticação. Visualiza cardápio e faz pedidos.

Administrador / Funcionário — acesso autenticado via JWT. Gerencia pedidos e visualiza operação.

Escopo do MVP

Cardápio público, realização de pedidos, acompanhamento por localStorage, painel admin com gestão de status.

Fora do escopo: gateway de pagamento, notificações push, login de cliente, analytics avançado.

Fluxo do Cliente

Acessa site → Vê cardápio → Monta pedido → Informa dados → Finaliza → Acompanha status

Fluxo do Administrador

Faz login → Recebe JWT → Vê pedidos → Atualiza status → Imprime / Entrega
Seção 2
Requisitos Funcionais
Funcionalidades que o sistema deve obrigatoriamente oferecer

Módulo: Cardápio (público)

RF01
Listar categorias: o sistema deve exibir as categorias disponíveis (ex: pizzas, sorvetes, bebidas).
RF02
Listar produtos por categoria: ao selecionar uma categoria, o sistema exibe os produtos com nome, descrição, preço e imagem (opcional).
RF03
Exibir disponibilidade: produtos marcados como indisponíveis não podem ser adicionados ao pedido e devem indicar visualmente essa condição.

Módulo: Pedido (público)

RF04
Montar carrinho: o cliente pode adicionar, remover e alterar a quantidade de itens antes de finalizar.
RF05
Informar dados do cliente: nome completo, número de telefone e endereço de entrega são obrigatórios.
RF06
Selecionar forma de pagamento: o cliente escolhe entre Pix, Cartão ou Dinheiro. Se dinheiro, pode informar o valor para troco.
RF07
Finalizar pedido: ao confirmar, o pedido é registrado no banco com status pendente e o cliente recebe um identificador de rastreamento.
RF08
Acompanhar status: usando o ID armazenado em localStorage, o cliente pode consultar o status atual do pedido.

Módulo: Administração (protegido)

RF09
Autenticação: o administrador faz login com e-mail e senha, recebendo um token JWT com expiração configurável.
RF10
Listar pedidos: o painel exibe todos os pedidos ordenados por data, com filtro por status.
RF11
Visualizar detalhes do pedido: nome do cliente, telefone, endereço, itens, total, forma de pagamento e valor de troco se aplicável.
RF12
Atualizar status do pedido: o admin pode avançar o status entre os estados definidos.
RF13
Gerenciar cardápio: CRUD de categorias e produtos (nome, descrição, preço, disponibilidade).
RF14
Imprimir pedido: o sistema oferece uma view de impressão com os dados completos do pedido formatados.
Seção 3
Requisitos Não Funcionais
Atributos de qualidade que definem como o sistema deve se comportar
RNF01 Performance
Resposta da API em menos de 300ms para 95% das requisições. Cardápio com cache — leitura não deve bater no banco a cada request.
RNF02 Escalabilidade
Arquitetura serverless (Vercel/Railway) escala horizontalmente sob demanda, sem necessidade de provisionamento manual.
RNF03 Segurança
Senhas armazenadas com bcrypt (fator ≥12). JWTs assinados com HS256, expiração de 8h. Rate limiting nos endpoints públicos.
RNF04 Disponibilidade
SLA mínimo de 99,5% via provedor gerenciado. Banco de dados com backup automático diário.
RNF05 Usabilidade
Interface responsiva (mobile-first). Fluxo de pedido concluível em menos de 2 minutos por um usuário leigo.
RNF06 Manutenibilidade
Código organizado em camadas (Clean Architecture). Cobertura de testes unitários ≥70% nas regras de negócio.
RNF07 Rastreabilidade
Logs estruturados (JSON) com ID de correlação por requisição. Integração com Sentry para erros de produção.
RNF08 Conformidade
LGPD: dados do cliente (nome, telefone, endereço) retidos por no máximo 90 dias após entrega, com possibilidade de deleção.
Seção 4
Modelagem do Banco de Dados
Entidades, relacionamentos e decisão arquitetural

Banco escolhido: PostgreSQL — relacionamentos bem definidos (pedido → itens → produtos), transações ACID para integridade do pedido, suporte nativo a JSON para flexibilidade futura, e excelente suporte serverless via Supabase ou Neon (connection pooling via PgBouncer).

Entidades

TabelaResponsabilidadeCampos principais
categoriesAgrupa os produtos do cardápioid, name, slug, active, sort_order
productsItem individual do cardápioid, category_id, name, description, price, image_url, available
ordersCabeçalho do pedidoid, tracking_code, customer_name, phone, address, payment_method, change_amount, status, total, created_at
order_itemsItens do pedido (snapshot de preço)id, order_id, product_id, product_name, unit_price, quantity, subtotal
admin_usersUsuários da área administrativaid, name, email, password_hash, role, active, last_login

Decisões de Design

◆
Snapshot de preço em order_items: os campos product_name e unit_price são copiados do produto no momento do pedido. Assim, alterar o preço no cardápio não afeta pedidos anteriores — integridade histórica garantida.
◆
tracking_code em orders: UUID ou código legível (ex: PED-A3F9) retornado ao cliente para acompanhar o status sem expor o ID interno.
◆
Normalização 3FN: order_items referencia products mas mantém cópia desnormalizada do preço. Isso é uma violação intencional e documentada para integridade histórica.
◆
change_amount nullable: o valor do troco só existe quando payment_method = 'cash'. Reforçado por constraint CHECK no banco.

Relacionamentos

Category 1:N Product — uma categoria possui muitos produtos

Order 1:N OrderItem — um pedido possui muitos itens

Product 1:N OrderItem — um produto aparece em muitos itens (leitura, FK opcional para deleção lógica)

Seção 4b
DDL — Create Tables
Scripts SQL prontos para produção com constraints e índices
-- Extensões necessárias
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

-- Categorias do cardápio
CREATE TABLE categories (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        VARCHAR(100) NOT NULL,
  slug        VARCHAR(100) NOT NULL UNIQUE,
  active      BOOLEAN NOT NULL DEFAULT true,
  sort_order  INTEGER NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Produtos do cardápio
CREATE TABLE products (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  category_id  UUID NOT NULL REFERENCES categories(id) ON DELETE RESTRICT,
  name         VARCHAR(200) NOT NULL,
  description  TEXT,
  price        NUMERIC(10,2) NOT NULL CHECK (price >= 0),
  image_url    TEXT,
  available    BOOLEAN NOT NULL DEFAULT true,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_available ON products(available);

-- Status possíveis do pedido
CREATE TYPE order_status AS ENUM (
  'pending',
  'confirmed',
  'preparing',
  'out_for_delivery',
  'delivered',
  'cancelled'
);

-- Formas de pagamento aceitas
CREATE TYPE payment_method AS ENUM ('pix', 'card', 'cash');

-- Pedidos
CREATE TABLE orders (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tracking_code   VARCHAR(20) NOT NULL UNIQUE,
  customer_name   VARCHAR(200) NOT NULL,
  phone           VARCHAR(20) NOT NULL,
  address         TEXT NOT NULL,
  payment_method  payment_method NOT NULL,
  change_amount   NUMERIC(10,2),
  status          order_status NOT NULL DEFAULT 'pending',
  total           NUMERIC(10,2) NOT NULL CHECK (total > 0),
  notes           TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- Troco só faz sentido para pagamento em dinheiro
  CONSTRAINT chk_change_cash ONLY CHECK (
    (payment_method = 'cash') OR (change_amount IS NULL)
  )
);

CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_tracking ON orders(tracking_code);
CREATE INDEX idx_orders_created_at ON orders(created_at DESC);

-- Itens do pedido (snapshot imutável)
CREATE TABLE order_items (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id      UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id    UUID REFERENCES products(id) ON DELETE SET NULL,
  product_name  VARCHAR(200) NOT NULL,  -- snapshot
  unit_price    NUMERIC(10,2) NOT NULL, -- snapshot
  quantity      INTEGER NOT NULL CHECK (quantity > 0),
  subtotal      NUMERIC(10,2) NOT NULL
);

CREATE INDEX idx_order_items_order ON order_items(order_id);

-- Usuários administrativos
CREATE TABLE admin_users (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name           VARCHAR(200) NOT NULL,
  email          VARCHAR(255) NOT NULL UNIQUE,
  password_hash  TEXT NOT NULL,
  role           VARCHAR(50) NOT NULL DEFAULT 'staff',
  active         BOOLEAN NOT NULL DEFAULT true,
  last_login     TIMESTAMPTZ,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Atenção: nunca armazene a senha em texto puro. O campo password_hash deve receber o resultado de bcrypt.hashpw(password, bcrypt.gensalt(rounds=12)). A aplicação é responsável pelo hash — o banco nunca vê a senha original.

Seção 5a
Arquitetura do Backend
Estrutura de pastas e padrão arquitetural adotado

Padrão: Clean Architecture (simplificada) — separação explícita entre domínio, casos de uso, infraestrutura e apresentação. Permite trocar banco ou framework sem reescrever regras de negócio.

app/ ├── domain/ # Entidades e regras puras │ ├── entities/ │ │ ├── order.py │ │ ├── product.py │ │ └── admin_user.py │ └── exceptions.py # DomainException, etc │ ├── use_cases/ # Orquestração da lógica │ ├── orders/ │ │ ├── create_order.py │ │ ├── get_order_status.py │ │ └── update_order_status.py │ ├── menu/ │ │ └── get_menu.py │ └── auth/ │ └── authenticate_admin.py │ ├── infrastructure/ # Detalhes externos │ ├── database/ │ │ ├── connection.py # SQLAlchemy engine │ │ ├── models.py # ORM models │ │ └── repositories/ │ │ ├── order_repo.py │ │ └── product_repo.py │ └── security/ │ ├── jwt_handler.py │ └── password_hasher.py │ ├── api/ # Camada FastAPI │ ├── v1/ │ │ ├── public/ │ │ │ ├── menu.py │ │ │ └── orders.py │ │ └── admin/ │ │ ├── auth.py │ │ ├── orders.py │ │ └── products.py │ ├── dependencies.py # get_db, get_current_admin │ └── schemas/ # Pydantic request/response │ ├── order_schemas.py │ └── product_schemas.py │ ├── core/ │ ├── config.py # Settings (pydantic-settings) │ └── logging.py # Structured logging setup │ └── main.py # FastAPI app factory

Camadas e Responsabilidades

◆
domain/ — entidades Python puras, sem dependência de FastAPI ou SQLAlchemy. Contém as regras de negócio que nunca mudam.
◆
use_cases/ — orquestra domínio e infraestrutura. Um caso de uso por funcionalidade. Testável sem banco real.
◆
infrastructure/ — implementações concretas: SQLAlchemy, bcrypt, PyJWT. Repositórios abstraem o banco.
◆
api/ — routers FastAPI, schemas Pydantic, injeção de dependências. Apenas converte HTTP ↔ use cases.

Seção 6
Regras de Negócio
Validações e restrições que o sistema deve enforçar

Pedidos

◆
Pedido não pode ser vazio: a lista de itens deve conter ao menos 1 item com quantidade ≥ 1.
◆
Produto disponível: apenas produtos com available = true podem ser adicionados ao pedido. Validação no use case, não somente no frontend.
◆
Forma de pagamento obrigatória: o campo payment_method é obrigatório e deve ser um dos valores permitidos (pix, card, cash).
◆
Troco exclusivo para dinheiro: change_amount só é aceito quando payment_method = 'cash'. Se informado para outro método, retorna 422. Se informado para dinheiro, deve ser ≥ ao total do pedido.
◆
Total calculado no backend: o frontend envia apenas os IDs e quantidades. O preço é lido do banco no momento do pedido — o cliente não envia o valor.

Status do Pedido

A transição de status segue uma máquina de estados linear. Não é permitido regredir ou pular etapas:

pending → confirmed → preparing → out_for_delivery → delivered

De qualquer estado antes de out_for_delivery, é permitido ir para cancelled.

Autenticação Admin

◆
Lockout após falhas: após 5 tentativas de login incorretas em 10 minutos, bloquear temporariamente por 15 minutos (implementar via rate limiting + Redis ou solução simples com flag no banco).
◆
Expiração do token: tokens expiram em 8h. O frontend deve renovar via refresh token (pode ser implementado em fase 2) ou redirecionar para login.

Cardápio

◆
Produto com pedido pendente: um produto não pode ser deletado se existir em pedidos com status ativo. Use deleção lógica (available = false).
◆
Preço mínimo: o preço de um produto deve ser > 0.

Serverless — Estratégias

◆
Cold start: o maior problema do serverless Python. Mitigue mantendo o requirements.txt enxuto e usando Mangum como adapter (AWS Lambda) ou deploy direto no Railway/Render que mantém instâncias quentes.
◆
Connection pooling: banco serverless como Neon usa PgBouncer internamente. Não crie uma nova conexão por request — use async_sessionmaker do SQLAlchemy com pool gerenciado.
◆
Cache do cardápio: o menu muda raramente. Use fastapi-cache2 com backend Redis (ou in-memory para MVP) com TTL de 60s. Invalide ao salvar produto.
